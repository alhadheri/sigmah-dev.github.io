Summary 0000634: Contacts database (aka "CRM")
Revision 2014-09-01 15:16 by osarrat
Additional Information This feature should add a new concept inside Sigmah in addition to Project and Orgunit: Contact.

A contact can be of 2 types: an Individual or an Organisation.
Contacts are as flexible as Projects and Orgunits: all contacts must be created from Contact Models. And the admin can create as many Contact Models as desired. A Contact Model must be either from the Individual or the Organization type. Each Contact Model can have as many fields of characteristics as desired.

This features adds a new field type: Contact relationship. This field can be configured to accept contacts only from a specific type (individual/organisation) or one or several contact models. This field can also be configured to accept a freely defined limit of number of contacts. This field is available for Project, Orgunit and Contact models. For Contact model, an additionnal optional can be set if the accepted contacts are only organisations for the field: "is membership".

An organisation contact page will have up to 4 sub-tabs: Characteristics, Child organisations, Members, Other relationships. The "Child organisations" and "Members" sub-tab will be filled with contact list of all contacts having a "membership" link with the organisation. If no organisation membership is possible, there will be no "Child organisation" sub-tab. Same for no "Members" if no individual membership.
The "Other relationships" sub-tab will list all other relationships with all contacts, projects and orgunits.

An individual contact page will look as an organisation contact page, but with only 2 sub-tabs: "Characteristics" and "Relationships".


This feature can be used with the Groupe iterations feature to be able to produce richly detailed relationships.

This "Export all" should export in OpenDocumentSpreadsheet/Excel format, and potentially also in ANSI SQL so that people will have a full relation database easily queriable.

In terms of rights:
- each contact is attached to a unique organisational unit
- the contact is visible by all users having the right to see its parent organisational unit
- a contact can contain fields having a privacy group, and privacy group fields should behave as privacy groups fields for projects or orgunits
- if a user is attached to several orgunits (see 0000243), its Contacts table on the dashboard will have its content updated according to the checked status of the "Display only my main organisational unit" checkbox: if checked (by default), only the contacts direct or indirect children of the main orgunit will be displayed, and if not all contacts direct or indirect children of all orgunits attached to the user will be displayed


This feature can be used with the Groupe iterations feature to be able to produce richly detailed relationships.
