Summary0000927: Project model to data framework mapping screen
DescriptionIn order to implement data framework by project model, a new tab "Data frameworks" must be added in project model edition screen.

On the left side of this new tab, there should be a foldable list of all "Available" data frameworks. This list will be a table with three columns: "Data framework" (each data framewor name will be an hyperlink opening the right side of the screen), "Implemented" (Implemented/No/Not Yet) and "Comment for non-implementation". By default, the value of "Implemented" is "Not Yet". Until all "Implemented" status are set to "Implemented" or "No", the project model cannot be set to "Available" status. If the data framework is made Available after a project model is itself set as Available, the Implemented value will be set to No by default with this Comment for non-implementation: "Data framework set as Available after project model was set as Available". A "Won't implement" button is available above the list: once a row in the list is selected, the button is enabled and if clicked it will open the "Data framework non-implementation popup" with a single mandatory field: "Comment for non-implementation", and two buttons: "Cancel" and "Confirm". Once the admin clicks on the name of a data framework, the list is folded and all the content of the tab is filled with the data framework mapping panel.

On the right ride of this tab, there is the "Data framework mapping" panel. This panel is filled with a big table listing all groups of elements and all data elements of the data framework in a hierarchical way. The table has several columns: "Element" (title of the group or data element), "Type" (type of data for data elements, or unique/iterative for groups), "Mapped to" ('constant' / name_of_field for data element, 'project model' / 'linked projects' / orgunit_model_name / group_name field for groups). And each row in the table is ended with an "Edit mapping" button.

If a group of elements is iterative, there is no default mapping rule: the "Mapped to" column is empty by default. As a consequence, the elements and groups it contains cannot have their mapping edited until a mapping rule is specified for it by clicking on its "Edit mapping" button. For groups of element having a unique instance, the default mapping rule is 'project model', so the data elements and groups they contain may be edited even if the default mapping rule is not edited.
Clicking on the "Edit mapping" button for an iterative group of element opens the "Group of element mapping" popup in which the admin can choose among the following mapping rules:
- for unique instance group of elements: single mapping rule / multiple mapping rules / multiple mapping rules with linked projects / multiple mapping rules with organisational unit )
- for iterative group of elements: multiple mapping rules / multiple mapping rules with linked projects / multiple mapping rules with organisational unit / project model iterative group / project model iterative group with parent filter

Depending on the chosen mapping rule, the lower part of the popup will change. In case of "multiple mapping rules" and "multiple mapping rules with linked projects" and "multiple mapping rules with organisational unit", there will be a "List of mapping rules" table filled by default by a rule called "Rule 0000001" which has priority 1. This table can be modified with a group of "New rule" / "Delete rule" / "Higher priority" / "Lower priority" buttons. If it is a "multiple mapping rules with linked projects", this list will be an ordered table with a project model name specified for each mapping rule. If it is a "multiple mapping rules with organisational unit", this list will be an ordered table with a orgunit model name specified for each mapping rule, and the orgunit to select: either "parent organisational unit" or "top organisational unit". The rule names can be modified in the list, and their name will be saved by clicking on the "Save" button at the very bottom part of the popup. For non-iterative groups of element, the rule which will be selected in case of multiple rules is the first rule for which there is a match: for example the first rule for which the corresponding orgunit with the right orgunit model and with the right role (parent for intance) is found.
In case of "project model iterative group", the bottom part will be a drop-down list containing all iterative groups of the project model. In case of "project model iterative group with parent filter", three informations will have to be specified: "Mapped group of field" through a drop-down list with all groups of field of the project model, "Parent iterative group" through a drop-down list containing all iterative groups of the project model, "Mapped group number field used as parent iterative group iteration selection" through a drop-down list containing all number fields of the mapped group of field. If there is no number field in the mapped group of field, the system will return an error when the mapped group of field will be selected.

For data element mapping rule, the edition will be vary according to its group of elements mapping: clicking on the data element "Edit mapping" button will have a different effect according the mapping rule chosen. In case of "multiple mapping rules" or "multiple mapping rules with linked projects" or "multiple mapping rules with organisational unit", a popup will open first asking to select the mapping rule to use. If it is "multiple mapping rules with linked projects", the fields available in the "Data element mapping" will only be fields of the project model set in the mapping rule. If it is "multiple mapping rules with organisational unit", the fields available in the "Data element mapping" will only be fields of the orgunit model set in the mapping rule. In case of "project model iterative group" and of "project model iterative group with parent filter", the fields available for mapping in the "Data element mapping" popup will only be the fields of the mapped group of fields. In the end, clicking on the "Edit mapping" button for data element always ends by opening the "Data element mapping" popup. If the group of element mapping rule is "single mapping rule", the "Edit mapping" button of the data element will directly open the "Data element mapping" popup.
On top of the "Data element mapping" popup, there is a "Mapping type" drop-down list with following values: mapping to a constant / mapping to a field. In case of "mapping to a constant", the rest of the popup will only be a mandatory "Constant" text field where the admin can enter any text or number matching the data element type restriction. In case of "mapping to a project field", there will be above the now optional "Constant" text field a "Field" drop-down list. Several additional options are possible in this case: if the field is text or list of choice field, it is possible to enable a "Part extraction filter" for which the admin has to specify the separator (can be any string) and if the part to extract is before or after the separator; if a constant is given in addition to the field, the admin can enable the "Constant concatenation" option and choose if he wants the constant to be concatenated before or after the value (or extracted part of the value) to produce the value sent has value of the data element; if the selected field is a Country field, an additional option appears which gives the possibility to map the iso2 code of the country instead of its name.
