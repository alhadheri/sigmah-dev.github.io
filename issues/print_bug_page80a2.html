Summary0000382: Moving top orgunit blocks the application
DescriptionIn the Organization Unit administration screen, if the administrator decides to move the top OrgUnit, he gets no immediate feedback of his action. But if he reloads the screen manually, he gets an error message in the OrgUnit administration screen, and he won't be able anymore to load even the main Dashboard screen. As the result, all users of the organization will always be blocked.

More globally, there is no way to change the top orgunit. The fixture proposed for this bug is then to change the behavior of the "Move" action for the top orgunit. Instead of asking for an orgunit to be moved under, if the selected orgunit is the top one, the popup will ask specifically for an orgunit to be switched with. Thus, there will always remain an orgunit at the top of the organigram of the organization.
Steps To ReproduceIn a freshly created organization context:
1. create an orgunit model and set it as "Available"
2. create a direct child orgunit "desk" of the "emptyHQ" orgunit
3. create a child orgunit "mission" of the child "desk"
4. try to move "emptyHQ" as child of "mission"
=> you get then a confirmation message informing you that the move has been performed successfully, but if you refresh your screen the list of orgunits won't be displayed anymore, and you will get a server error if you try to open the main dashboard.

For information: a new message informing the user that he cannot move an orgunit as a child of one of its children appears if you try to move "emptyHQ" as child of "desk". (but it doesn't appear of you try the same with "mission" as destination.
