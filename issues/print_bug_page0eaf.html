Summary0000510: Interruption project phases
DescriptionSometimes, the project cycle cannot be followed as expected. The project can be freezed at some point, cancelled, or closed earlier for example.
The phases workflow described in a project model should be able to represent this reality. Each phase should be able to have another type of successors which would be "interruption successors".
When a phase has an "interruption successor", the user with a new global privilege "interrupt project phase" can see a new button : "Interrupt phase". By clicking on this button, he will see the list of interruption phase linked to this phase as interruption successors. Going to an interruption successor doesn't freeze the current phase, it will just open to modifications the interruption phase selected by the user. Phases reached through this "Interrupt phase" button will have almost the same behavior as other phases : they can be closed only if all their mandatory fields are filled, and they can also have "interruption successors". Their only difference is that, if they are accessed as "interruption phase", when the user will be able to click on the "Close phase" button, he will only see 2 potential options: "go back to your interrupted phase 'XXXX'", or "set the whole project as 'Closed'" if this phase has as potential sucessor a new "Closed project status phase".
This "Closed project status phase" will be a phase present in the list of phases which cannot be removed, and which has always to be linked to at least one predecessor. This new special phase will give the ability to let the user close a project anytime by interruption if this phase is selected among the potential "interruption phase", or give several time when a process can close the project after ending a phase normally.

The Close/Open status should be available by default in the Project Table : closed projects will have a "Closed" label printed in place of the time consumption progress bar.
