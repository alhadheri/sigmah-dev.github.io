Summary0000934: Extended list of contacts
DescriptionSometimes, there is a need to store information about the relationship between a contact and a project (like the grade of the success at the end of a training). The following functional solution has been designed to reply to this need of extended list of contacts (the term "extended list of contacts" might be revised later on). In the future, there might be feature requests for extended list of documents, extended list of reports, etc., so the development of this feature should try as far as possible to take the potential development of those other features in consideration.

The iterative group feature should be modified to reply to this need. Only in contact model, in the group edition popup, two new fields should be added: a "used as extended list of contacts attributes" checkbox, and a "Contacts list field" message field. When creating a new group, if the "iterative group" checkbox is checked, the "used as extended list of contacts attributes" checkbox is enabled. If the admin clicks on this "used as extended list of contacts attributes" checkbox, he gets the following confirmation message: "Using a new iterative group as a attributes for an extended list of contacts mean that iteration will be able to be created for that iterative group only as new element in the linked contact list field. Are you sure to continue?". The rest of the creation and modification of the iterative group will be as usual.
In the project/contact/orgunit model where the admin wants to have his extended list of contacts, he will just have to fill a new specific property in the contact list field he wants to extend: "Iterative group used for extension". This property will be made of one drop-down list filled with all groups marked as "used as extended list of contacts attributes" and not yet linked to any contact list field. Each group will be presented as "'[GROUP_TITLE]' of [CONTACT_MODEL_NAME] contact model". The drop-down list will have a blank value which will be the default value. This new "Iterative group used for extension" property will have to be before the contact model and contact type contraints property, because if a group is selected to make an extended list of contacts, the content of those two constraints will be modified to the contact model and contact type of this model, and won't be modifiable anymore until this "Iterative group used for extension" is set back to blank.

For the end-user, in the project/orgunit/contact having this extended list of contacts, for each addition of a new contact, after having submitted the normal contact addition form, a popup will automatically: "Extended informations on contact relationship" which will reproduce exactly the content of the linked iterative group of fields. The user will have to fill all mandatory fields before closing the popup. And for each addition of a contact, this will naturally create a new iteration in the linked iterative group.
In the extended list of contact, there should be an "Edit extended informations" button for each contact. This button will open the same "Extended informations on contact relationship" popup.
In the iterative group in the contact page, there will be no "Rename iteration"/"Remove iteration" buttons, neither the new iteration "+" tab. But there will be a special "Open linked [project/organisational unit/contact] [OBJECT_FULL_NAME]" button in the iterative group toolbar.

The importation mapping process has also to be modified to allow the importation of contacts will all details requested in the extended informations edition popup. If some mandatory information are missing in the source imported file for some contacts, the "Extended informations on contact relationship" popup will open pre-filled with available informations and will be able to be closed only when the user will have filled all mandatory informations.
