Summary 0000530: New type of orgunit/project field: computation field
Revision 2013-02-25 15:08 by osarrat
Description A new type of field should be added in project/orgunit model: computation field. This field is similar to the organizational unit aggregation field but will aggregate information only on fields of the same orgunit/project model. Computation operations requested: sum, ratio, difference, multiplication. Computation can be done on more than 2 fields of the same model.
As part of the computation, a constant can be added (and for dates, it can be either a specific date or today), or global variables (like total amount of funding from all sources of funding, total funded from all funded projects)

Revision 2015-11-23 15:46 by osarrat
Additional Information The detailed behavior of the feature has been imagined as follows :
A. Modifications in the server database schema:
To make this feature work, the following modifications will be made on the server database schema:
- computation field will be a new kind of "flexible element", which will store two informations: its latest computed value, and in a text format the formula used for computation with fields part of computation represented as "$ID_OF_FIELD" ($2460, $13986, etc.)
- a new characteristic of field is added, the "field code" which will be used in computation fields formulas
- fields part of computation are flagged differently and associated to the computation field they are part of

B. Administration area
- In the Field popup, a new field "Field code" will give the possibility to edit the default code given to the field. Each field has by default a field code in the format "field001", "field045", etc. The numbering of this default field codes is done from the first created fields in the model.
- When creating or editing the computation field in a project/orgunit model, a formula must be entered using the field codes to represent the fields. The list of all fields of the model is available to help the administrator to identify and get the codes of the fields he wants to use, and this list is editable on the field code. When saving the modifications of the computation field, the system checks that the entered formula is valid and that all field codes used match the most up-to-date field codes in database. Currently, it won't be accepted to make computation field using other computation fields.
- When deleting or disabling a field part of a computation in a project/orgunit model, a warning message will inform the administrator that : "The field you want to disable/delete is used in computation field(s) NAME_OF_THE_COMPUTATION_FIELD[, list of computation fields where the field is used]. If you disable/delete it without modifying the formula of those computation field(s), the computed values might be aberrant. Are you sure you want to continue?"

C. For the end-user
- When editing a field part of the computation, the computed value is changed (without controlling the optional limits) when the edition of the field is ended so that the user can be informed of the computed value.
- When saving ONLINE a modification of one or several fields part of the computation, the server computes the new aggregated value based on the latest values stored in the server database and the modifications made by the user : if there is a limit and if this limit is exceeded, the modification made by the user is rejected and he gets a detailed warning message informing him that "Your latest modification on field(s) FIELD1, FIELD2, [List of fields modified and part of the computation] cannot be saved because it makes the computed value of NAME_OF_THE_COMPUTATION_FIELD [below/greater] than EXCEEDED_LIMIT. For your information, here are the latest values of all fields used to compute NAME_OF_THE_COMPUTATION_FIELD: [table with columns: Title / Value / Latest modification (datetime) / Author ]"
- When saving OFFLINE a modification of one or several fields part of the computation, the software on the client-side computes the new aggregated value based on the latest values stored in its offline data base and save always anyway the modification but if a limit is exceeded returns as well the following message: "According the values collected at different times in your offline data base, your latest modification on field(s) FIELD1, FIELD2, [List of fields modified and part of the computation] would make the computed value of NAME_OF_THE_COMPUTATION_FIELD [below/greater] than EXCEEDED_LIMIT. When you will synchronize your offline data base with the server, this modification might then be rejected if it makes the value computed with the other server-side values exceed a limit."
- When synchronizing an offline data base with modification of a field part of a computation, control of the aggregated value will be made and the modification might be rejected as if the edition was made online and rejection can join the final synchronization feedback and email.


IN THE FUTURE
In an extended version of this feature, dates will be able also to be taken into account, and a constant could be added as part of the computation (and for dates, it can be either a specific date or today), or global variables (like total amount of funding from all sources of funding, total funded from all funded projects).
